3. 发现的问题

3.0 迁移文件与代码不一致 🔴 严重

3.0.1 迁移文件缺失表定义

问题描述: 迁移文件 migrations/versions/20260321_001_initial_schema.py 仅定义了 4 个表，但代码中实际使用了 10 个表。

3.1 索引问题

3.1.1 索引过多

问题描述: daily_messages 表有 12 个索引，过多的索引会：

  • 降低 INSERT/UPDATE 性能
  • 增加存储空间
  • 增加维护成本

3.1.2 缺少关键索引

表 列 查询场景 状态
daily_messages feishu_conversation_id 会话历史查询 缺失
daily_messages conversation_id 会话关联 缺失
sessions expires_at 过期会话清理 缺失

3.2 数据类型问题

3.2.1 日期/时间存储为字符串

-- 当前定义
date VARCHAR NOT NULL
timestamp VARCHAR

-- 建议定义
date DATE NOT NULL
timestamp TIMESTAMP

3.2.2 布尔值使用 INTEGER

-- 当前定义
is_active INTEGER DEFAULT 1
is_group_chat INTEGER

-- PostgreSQL 建议定义
is_active BOOLEAN DEFAULT TRUE
is_group_chat BOOLEAN

    ON sessions(is_active, expires_at);

4.2 方案二：添加外键约束

# 为 quota_usage 表添加外键
def upgrade():
    op.create_table(
        'quota_usage_new',
        sa.Column('id', sa.Integer(), primary_key=True),
        sa.Column('user_id', sa.Integer(), sa.ForeignKey('users.id', ondelete='CASCADE'), nullable=False),
        sa.Column('date', sa.String(), nullable=False),
        sa.Column('period', sa.String(), server_default='daily'),
        sa.Column('tokens_used', sa.Integer(), server_default='0'),
        sa.Column('requests_used', sa.Integer(), server_default='0'),
        sa.Column('created_at', sa.TIMESTAMP(), server_default=sa.text('CURRENT_TIMESTAMP')),
        sa.Column('updated_at', sa.TIMESTAMP(), server_default=sa.text('CURRENT_TIMESTAMP')),
    )

    op.execute("INSERT INTO quota_usage_new SELECT * FROM quota_usage")

    op.drop_table('quota_usage')
    op.rename_table('quota_usage_new', 'quota_usage')

    op.create_index('idx_quota_user_date', 'quota_usage', ['user_id', 'date'])

5. SQLite vs PostgreSQL 对比

5.1 功能对比

功能 SQLite PostgreSQL
并发写入 ❌ 单写者锁 ✅ MVCC 多写者
JSON 支持 ⚠️ 基础 ✅ JSONB + GIN 索引
全文搜索 ⚠️ FTS5 ✅ 内置 tsvector
窗口函数 ⚠️ 有限 ✅ 完整支持
行级锁 ❌ ✅
外键约束 ⚠️ 可选 ✅ 强制
复制/HA ❌ ✅ 流复制
分区表 ❌ ✅

5.2 适用场景

场景 SQLite PostgreSQL
单机/个人使用 ✅ 推荐 过度设计
开发/测试环境 ✅ 推荐 ✅ 推荐
小团队 (<10 人) ✅ 可接受 ⚠️ 可选
中团队 (10-50 人) ⚠️ 性能瓶颈 ✅ 推荐
生产环境/多租户 ❌ 不推荐 ✅ 推荐
需要高可用 ❌ 不支持 ✅ 推荐

🎯 结论

建议迁移到 PostgreSQL，特别是如果：

  • 用户数 > 10
  • 需要多实例部署
  • 需要高可用
  • 需要复杂查询/分析
